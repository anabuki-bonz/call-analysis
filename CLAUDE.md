# CLAUDE.md — コールセンター日次分析

---

# 目的

コールセンターの日次データを自動取得・分析・通知する。

---

# 実行ルール

## 基本制約

* 指定STEPのみ実行する
* 新規コード生成禁止
* 自由分析禁止
* 追加提案禁止
* 推測を事実として記載しない
* 不明時は「確認できません」と記載
* 出力は必要最小限
* notify.py はSTEP4で最後に1回のみ実行

## 起動トリガー

| 指示 | 対象 |
|------|------|
| 分析して | 前日 |
| M/D分析して | 指定日 |
| M/D〜M/D分析して | 指定範囲（1日ずつ個別実行） |

複数日指定時は各日ごとに STEP1〜STEP2 を実行し、日別にレポートを生成する。全日完了後に STEP3・STEP4 を各1回実行する。

## 実行手順

各STEPは1回のみ実行する。

### STEP1: データ取得

```bash
cd "G:\マイドライブ\Claude Code\call-analysis"
python -X utf8 call_center/fetch_sheets.py
```

### STEP2: 分析実行

```bash
python -X utf8 call_center/analyze.py YYYY-MM-DD
```

### STEP3: ダッシュボード公開

```bash
cd "G:\マイドライブ\Claude Code\call-analysis"
git add .
git commit -m "feat: update dashboard YYYY-MM-DD"
git push
```

* commit メッセージの日付は分析対象日に置き換える
* push 完了を確認したら次のステップへ

### STEP4: 通知送信

```bash
sleep 120
python -X utf8 call_center/notify.py YYYY-MM-DD
```

* STEP3 の push exit code 0 を確認してから実行する
* push 後 120秒待機してから notify.py を実行する（GitHub Pages 反映待ち）
* notify.py は最後に1回のみ実行する
* notify.py の重複実行禁止
* exit code 0 を確認したら終了

## 完了条件

STEP4 の exit code 0 を確認したら完了。

完了後は「日次分析が完了しました。」のみ返答する。追加実行・追加分析・補足説明は行わない。

## Python実行ルール

* 常に `python -X utf8` で実行する（Windows cp932誤検知防止）
* 実行対象は既存.pyファイルのみ
* 新規コード生成・一時コード実行・ヒアドキュメント禁止
* 失敗時は再実行せず終了

# 定義

## KPI目標

応答率 = 応答数 ÷ 着信数

| 区分 | 目標 |
|------|------|
| クライアント向け | 90%以上 |
| 社内管理 | 95%以上 |

## 指標定義

| 指標 | 定義 |
|------|------|
| 着信数 | 営業時間内IN着信総数 |
| 応答数 | オペレーター接続成立件数 |
| 未応答数 | 接続前終了件数（着信数 − 応答数） |
| CPH | 応答数 ÷ 勤務時間（稼働時間シートF列・小数第1位四捨五入） |
| 平均通話時間 | 応答通話平均秒数 |
| 処理可能数 | 稼働時間 × 10 |
| 適正人員乖離率 | (着信数 − 処理可能数) ÷ 着信数 |

## 用語統一

使用する用語: 着信数 / 応答数 / 未応答数 / 応答率

使用禁止: 入電数 / 受電数 / 放棄数 / 呼損 / 取得率

## 色分け基準

* 🟢 応答率90%以上
* 🟡 応答率80〜89%
* 🔴 応答率80%未満

# 分析ロジック

## 稼働判定

* 分析対象: Div. = IN のOPのみ
* 稼働時間: 出勤〜退勤のうち休憩時間を除外した時間
* 禁止: 通話ログ（CDR/Zoiper）による稼働判定、ログイン有無のみの判定

## 未応答分類

| 分類 | 条件 |
|------|------|
| Ⅰ 休憩中 | OP存在 = true ／ 休憩中 = true ／ 放棄Zoiper記録 = false |
| Ⅱ 混雑 | OP存在 = true ／ 放棄Zoiper記録 = true |
| Ⅲ 範囲外 | OP存在 = true ／ 放棄Zoiper記録 = false |
| Ⅳ 稼働なし | 稼働実績 = false または 全員休憩 = true |

## ランキングルール

**着信数ランキング（上位5件）**

* 着信数降順・同率時は応答率降順

**応答率ランキング（ベスト5 / ワースト5）**

* ベスト: 応答率降順・同率時は着信数降順
* ワースト: 応答率昇順・同率時は着信数降順

**特定案件ランキング**

* 案件名の先頭に [A] または [B] を含む（全角・半角どちらも対象）
* 応答率 < 90% を対象
* 応答率昇順・同率時は着信数降順

## 異常判定

| 比較 | 計算式 | 判定閾値 |
|------|--------|---------|
| 前日比 | (今日 − 前日) ÷ 前日 | abs ≥ 10% |
| 前週比 | (今日 − 前週) ÷ 前週 | abs ≥ 10% |
| 前月比 | (今日 − 前月) ÷ 前月 | abs ≥ 10% |

* 表示: 小数点第2位四捨五入（%表示）
* 比較元データ = 0 の場合: 「確認できません」と記載

## 分析範囲

実施: KPI算出 / 閾値判定 / 前日・前週・前月の独立異常検知 / 定義済み異常抽出

禁止: 恒常傾向断定 / OP個人批判 / 自由分析

# 分析フレームワーク

日次・週次・月次の運営分析において、単なる数値報告ではなく、なぜその結果になったのか・再発防止できるのか・今後も継続する要因なのか・改善アクションは何かを明確にするための基準。

## KPI評価基準

| 評価 | 応答率 | 状態 | 主な確認事項 |
|------|--------|------|-------------|
| S | 98%以上 | − | 人員過多・着信予測下回り・稼働率低下・過剰コスト |
| A | 95〜98%未満 | 理想的 | 未応答発生時間帯・突発入電・稼働率適正性 |
| B | 90〜95%未満 | 要観察 | 未応答原因・人員不足・処理時間増加・案件集中 |
| C | 85〜90%未満 | 要分析 | 予測差異・シフト不足・欠勤・AHT増加・エスカレーション増加 |
| D | 85%未満 | 改善必須 | 人員不足・大量入電・障害・欠勤・教育不足・運用ミス |

## 自動判定項目（ログデータ根拠）

### 需要要因

* 着信数の前日比・前週比 ±10%以上の変動
* 時間帯別着信集中（1時間帯が全体の20%以上）

### 供給要因

* 時間帯別人員不足（必要人数 > 実績人数）
* 全体の適正人員乖離率（超過/余剰 ±10%以上）

### 生産性要因

* 平均CPH低下（8件/h未満）または異常上昇（12件/h以上）
* CPHの前日比 ±20%以上の変動
* 平均通話時間5分超過または前日比 ±20%以上の変動

## 要因分析

数値のみを根拠に短絡的な結論を出さず、複数の可能性を検討した上で要因を考察する。

### 基本原則

割合変化と絶対値を必ず併用すること。

* 前週比+50%でも件数差が小さい場合は影響を限定的と評価する
* 前週比+5%でも件数差が大きい場合は影響ありと評価する

応答率のみで評価しない。以下の5観点から要因を整理する。

### 分析観点

| 観点 | 確認項目 |
|------|---------|
| 需要要因 | 着信増加・キャンペーン・メール配信・障害発生・季節要因・曜日要因 |
| 供給要因 | 人員不足・欠勤・遅刻・休憩集中・シフト設計 |
| 生産性要因 | AHT増加・ACW増加・保留時間増加・エスカレーション増加 |
| 品質要因 | FAQ不足・マニュアル不足・教育不足・クレーム増加 |
| システム・運用要因 | CTI障害・kintone障害・ネットワーク障害・AIツール障害 |

### 応答率評価時の注意

応答率100%を無条件に高評価しない。人員過多・着信減少・稼働率低下・過剰コストの可能性を検討する。

応答率低下を無条件に低評価しない。想定外の着信増加・キャンペーン・障害・クライアント施策の影響を検討する。

### 分析手順

1. 発生した事実を整理する
2. 要因仮説を最低3つ抽出する
3. 各仮説に対する反対仮説を検討する
4. 現時点で最も確からしい要因を整理する
5. 不足しているデータを明示する

### 仮説検証ルール

仮説は最低3つ挙げること（例: 人員不足 / 着信増加 / AHT増加）。

各仮説に対して反対仮説を提示すること。

> 仮説: 人員不足が発生した可能性がある
>
> 反対仮説: 人員数は十分だったが、AHT増加により処理能力が低下した可能性がある

### 出力形式

**確認された事実** — 事実のみ記載

**要因仮説** — 仮説① / 仮説② / 仮説③

**反対仮説** — 反対仮説① / 反対仮説② / 反対仮説③

**真因候補** — 確からしさが高い順に記載

**不足データ** — 追加確認が必要なデータを記載

### 禁止事項

以下のような根拠のない断定は禁止:

* 応答率が低いため人員不足である
* 忙しかったため未応答が増加した
* データが存在しない事項の断定

事実と推測を明確に区別する。不明な場合は「確認できない」と記載する。
